# Feature Specification: Inline Episode Registration for Presenters

**Feature Branch**: `003-presenter-episode-inline`
**Created**: 2025-01-14
**Status**: Draft
**Input**: User description: "Presenterを登録する際に、画面上で３つのエピソードと嘘かどうかのフラグを選択できるように修正してください。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Single-Form Presenter Registration with Episodes (Priority: P1)

ゲーム管理者が、プレゼンター1人とその3つのエピソード（2つのホント、1つのウソ）を1つのフォームで一度に登録できる。

**Why this priority**: このストーリーはMVPの核心機能です。現在の2ステップのプロセス（プレゼンター登録→エピソード追加×3）をワンステップに統合することで、ユーザーエクスペリエンスが大幅に改善されます。この機能だけで独立して価値を提供でき、すぐにユーザーに試してもらえます。

**Independent Test**: プレゼンター管理画面で、1つのフォームにニックネームと3つのエピソードを入力し、送信ボタンを押すと、プレゼンターと3つのエピソードが同時に作成されることを確認することでテストできます。

**Acceptance Scenarios**:

1. **Given** ゲーム管理画面にいる管理者、**When** プレゼンター追加フォームを開く、**Then** ニックネーム欄と3つのエピソード入力欄（各エピソードにテキストエリアとウソマーカーチェックボックス）が表示される
2. **Given** プレゼンター登録フォーム、**When** ニックネーム「田中太郎」、エピソード1「高校で野球部だった」（ホント）、エピソード2「猫を3匹飼っている」（ホント）、エピソード3「フランス留学経験がある」（ウソ）を入力して送信、**Then** プレゼンター「田中太郎」が3つのエピソードと共に登録され、プレゼンター一覧に表示される
3. **Given** フォーム入力中、**When** ウソマーカーを2つ以上のエピソードにチェックしようとする、**Then** 「ウソのエピソードは1つだけ選択してください」というバリデーションエラーが表示される
4. **Given** フォーム入力中、**When** ウソマーカーを1つもチェックせずに送信、**Then** 「ウソのエピソードを1つ選択してください」というバリデーションエラーが表示される
5. **Given** フォーム入力中、**When** エピソードのテキストエリアを空欄のまま送信、**Then** 「すべてのエピソードを入力してください」というバリデーションエラーが表示される
6. **Given** プレゼンター登録成功後、**When** プレゼンター一覧を確認、**Then** 新しく追加されたプレゼンターが3つのエピソード（2つのホント、1つのウソ）と共に表示される

---

### User Story 2 - Episode Character Count and Validation Feedback (Priority: P2)

各エピソード入力欄にリアルタイムの文字数カウンターとバリデーションフィードバックが表示される。

**Why this priority**: ユーザーが入力制限を理解しやすくなり、送信前にエラーを防ぐことができます。P1の基本機能が動作した後に、UX向上のために追加する価値があります。独立してテスト可能で、フォームの使いやすさを向上させます。

**Independent Test**: エピソード入力欄に文字を入力すると、各欄の下に「X/1000文字」というカウンターが表示され、1000文字を超えると赤色で警告が表示されることを確認できます。

**Acceptance Scenarios**:

1. **Given** プレゼンター登録フォーム、**When** エピソード1のテキストエリアに文字を入力、**Then** テキストエリアの下に「X/1000文字」というカウンターが表示される
2. **Given** エピソード入力中、**When** 1000文字を超える文字を入力、**Then** 文字数カウンターが赤色で表示され、「1000文字以下で入力してください」という警告が表示される
3. **Given** エピソード入力中、**When** 空欄のまま他のフィールドにフォーカスを移動、**Then** 「エピソードを入力してください」というエラーメッセージが表示される

---

### User Story 3 - Form Reset and Clear Functionality (Priority: P3)

プレゼンター登録フォームをリセットして、すべての入力内容をクリアできる。

**Why this priority**: ユーザーが誤入力した場合や、やり直したい場合に便利です。基本機能（P1、P2）が完成した後に、追加の利便性を提供します。

**Independent Test**: フォームに入力した内容を「クリア」ボタンで全て削除できることを確認できます。

**Acceptance Scenarios**:

1. **Given** プレゼンター登録フォームに入力内容がある、**When** 「クリア」ボタンをクリック、**Then** すべての入力欄（ニックネーム、3つのエピソード、ウソマーカー）が空になる
2. **Given** フォームリセット後、**When** 再度入力を開始、**Then** 正常に入力できる

---

### Edge Cases

- **エピソード数が3つ未満の場合**: 送信時に「3つすべてのエピソードを入力してください」というバリデーションエラーを表示する
- **ニックネームが空の場合**: 「ニックネームを入力してください」というバリデーションエラーを表示する
- **ニックネームが50文字を超える場合**: 「ニックネームは50文字以下で入力してください」というバリデーションエラーを表示する
- **エピソードのテキストが1000文字を超える場合**: 「エピソードは1000文字以下で入力してください」というバリデーションエラーを表示する（各エピソード個別に）
- **ウソマーカーが0個の場合**: 「ウソのエピソードを1つ選択してください」というバリデーションエラーを表示する
- **ウソマーカーが2個以上の場合**: 「ウソのエピソードは1つだけ選択してください」というバリデーションエラーを表示する
- **ネットワークエラー時**: 「登録に失敗しました。もう一度お試しください」というエラーメッセージを表示し、入力内容を保持する
- **セッションタイムアウト時**: 「セッションが切れました。ログインし直してください」というエラーメッセージを表示する
- **同時送信防止**: 送信ボタンをダブルクリックしても、1回のみ送信される（送信中はボタンを無効化）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、プレゼンター登録フォームにニックネーム入力欄を1つ表示しなければならない
- **FR-002**: システムは、プレゼンター登録フォームにエピソード入力欄を3つ表示しなければならない（各エピソードにテキストエリアとウソマーカーチェックボックス）
- **FR-003**: システムは、ニックネームが1文字以上50文字以下であることを検証しなければならない
- **FR-004**: システムは、各エピソードのテキストが1文字以上1000文字以下であることを検証しなければならない
- **FR-005**: システムは、3つのエピソードすべてが入力されていることを検証しなければならない
- **FR-006**: システムは、ウソマーカーが正確に1つだけチェックされていることを検証しなければならない（0個または2個以上の場合はエラー）
- **FR-007**: システムは、フォーム送信時にプレゼンターと3つのエピソードをアトミックに（全てが成功するか、全てが失敗するか）保存しなければならない
- **FR-008**: システムは、送信中はフォームの入力欄と送信ボタンを無効化しなければならない
- **FR-009**: システムは、登録成功時に成功メッセージを表示し、プレゼンター一覧を更新しなければならない
- **FR-010**: システムは、バリデーションエラー時に具体的なエラーメッセージを該当フィールドの近くに表示しなければならない
- **FR-011**: システムは、各エピソード入力欄に文字数カウンター（X/1000文字）を表示しなければならない
- **FR-012**: システムは、ニックネーム入力欄に文字数カウンター（X/50文字）を表示しなければならない
- **FR-013**: システムは、既存のエピソード個別追加機能を削除または非表示にしなければならない（プレゼンター登録後にエピソードを追加する機能は不要になる）

### Key Entities

- **Presenter**: プレゼンターを表すエンティティ（属性: id, gameId, nickname, episodes[], createdAt）
- **Episode**: エピソードを表すエンティティ（属性: id, presenterId, text, isLie, createdAt）
- **PresenterWithEpisodesInput**: プレゼンター登録時の入力データ（nickname: string, episodes: [{ text: string, isLie: boolean }] × 3）
- **AddPresenterWithEpisodesSchema**: Zodバリデーションスキーマ（ニックネーム、3つのエピソード、ウソマーカーの検証ルール）

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーがプレゼンター1人を登録するのに必要な画面遷移が0回（現在は複数回のエピソード追加フォーム表示が必要）
- **SC-002**: プレゼンター登録の平均時間が50%削減される（2ステッププロセスから1ステッププロセスへの変更により）
- **SC-003**: フォーム送信時のバリデーションエラー率が30%以下（リアルタイムフィードバックと文字数カウンターにより）
- **SC-004**: プレゼンター登録の成功率が95%以上（アトミックな保存処理により、部分的な失敗を防ぐ）
- **SC-005**: すべての単体テストとE2Eテストがパスする（フォームバリデーション、送信処理、エラーハンドリングのテストを含む）

## Technical Considerations *(optional)*

### Data Flow

**現在のフロー（2ステップ）**:
1. `PresenterForm` → `addPresenterAction` → `AddPresenter` use case → Presenter作成
2. `EpisodeForm` (3回) → `addEpisodeAction` → `AddEpisode` use case → Episode作成 × 3

**新しいフロー（1ステップ）**:
1. `PresenterWithEpisodesForm` → `addPresenterWithEpisodesAction` → `AddPresenterWithEpisodes` use case → Presenterと3つのEpisodeを同時作成

### Backend Changes Required

- **新規Zodスキーマ**: `AddPresenterWithEpisodesSchema`（nickname + episodes[3]のバリデーション）
- **新規Use Case**: `AddPresenterWithEpisodes`（プレゼンターと3つのエピソードをトランザクション的に作成）
- **新規Server Action**: `addPresenterWithEpisodesAction`（フォームデータを受け取り、use caseを実行）
- **既存機能の削除/非表示**: `EpisodeForm`コンポーネントと`addEpisodeAction`は削除またはレガシー扱いにする

### Frontend Changes Required

- **新規コンポーネント**: `PresenterWithEpisodesForm`（ニックネーム + 3つのエピソード入力欄）
- **新規Custom Hook**: `usePresenterWithEpisodesForm`（フォーム状態管理、バリデーション、送信処理）
- **既存コンポーネントの置き換え**: `PresenterManagementPage`で`PresenterForm`の代わりに`PresenterWithEpisodesForm`を使用

### Validation Rules

```typescript
// Zodスキーマの概要
AddPresenterWithEpisodesSchema = z.object({
  gameId: z.string().min(1),
  nickname: z.string().min(1).max(50),
  episodes: z.array(
    z.object({
      text: z.string().min(1).max(1000),
      isLie: z.boolean()
    })
  ).length(3) // 正確に3つ
}).refine(
  data => data.episodes.filter(e => e.isLie).length === 1,
  { message: "ウソのエピソードは1つだけ選択してください" }
)
```

### UI/UX Design Notes

- **レイアウト**: 縦に並べたフォーム（ニックネーム → エピソード1 → エピソード2 → エピソード3 → 送信ボタン）
- **視覚的な区別**: 各エピソードセクションを軽いボーダーで区切る
- **ウソマーカー**: ラジオボタンではなくチェックボックスを使用（1つだけ選択する制約はバリデーションで実装）
- **エラー表示**: インラインエラー（各フィールドの下）+ トップレベルエラー（フォーム全体のエラー）
- **成功フィードバック**: 送信成功時に「プレゼンターと3つのエピソードを登録しました！」というメッセージを表示

## Assumptions *(optional)*

- プレゼンター登録後にエピソードを追加・編集する機能は不要（この機能により、プレゼンターは常に3つのエピソードと共に作成される）
- 既存のプレゼンターにエピソードを後から追加する機能は削除される（新しいフローでは最初から3つ登録）
- ゲームの状態が「準備中」の時のみプレゼンター登録が可能（既存の制約を維持）
- InMemoryRepositoryを引き続き使用（トランザクション的な保存は、リポジトリメソッド内でロールバック処理を実装）

## Out of Scope *(optional)*

- プレゼンター編集機能（既存のプレゼンターのニックネームやエピソードを変更する機能は本機能に含まれない）
- エピソード数の可変化（常に3つ固定）
- ウソマーカーの複数選択（常に1つのみ）
- プレゼンター削除機能の変更（既存機能をそのまま維持）
